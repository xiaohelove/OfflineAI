1. 产品概述
==========

1.1 产品简介
----------

离线AI助手是一款面向设备维修技术人员的 Android 本地智能诊断应用。用户导入产品维修手册（PDF/TXT）后，应用基于本地大语言模型进行检索增强生成（RAG）问答。所有计算在设备本地完成，无需网络连接。

1.2 核心价值
----------

- 完全离线：模型与数据均在本地，适用于无网络覆盖的维修现场
- 数据安全：手册内容不上传云端，敏感技术文档不外泄
- 精准溯源：诊断建议引用手册原文编号，可追溯验证
- 即装即用：首次启动自动解压模型，冷启动时间小于 5 秒

3. 产品定位与目标
==========

3.1 产品定位
----------

面向无网络和涉密场景的本地 AI 知识库工具。用户导入 PDF 手册后，在完全离线的环境下获得基于原文的精准问答，适用于设备维修、野外作业、应急救援、涉密内网等场景。

3.2 产品目标
----------

目标一：打造可用的离线 RAG 问答体验
- RAG 检索 Top-5 准确率不低于 85%
- 回答引用可追溯至原文
- 单次推理端到端延迟不超过 120 秒（2B 模型）

目标二：降低使用门槛
- 首次启动时间不超过 40 秒
- PDF 导入至可查询时间不超过 30 秒
- 新用户首次问答成功率不低于 80%

目标三：确保稳定可靠
- 应用崩溃率低于 0.1%
- 模型加载成功率不低于 99%
- 无内存溢出导致的应用退出

4.3 场景详解
----------

场景一：野外求生应急查询

用户携带装载离线知识库（如野外生存手册、急救指南、装备说明书）的 Android 手机进入无信号区域。遇突发情况时，打开应用输入问题（如"被蛇咬伤如何处理""岩壁下降绳索固定方法"），应用在完全离线状态下完成检索和推理，返回基于手册原文的指导。

此场景对产品的核心要求：
- 零网络依赖，所有功能在飞行模式下正常运行
- 知识库支持用户自行导入任何 PDF 格式的野外生存手册或技术文档
- 响应结果必须引用原文，便于用户在紧张情况下核对原始资料

场景二：地质勘探现场技术支援

地质勘探人员在偏远山区作业，随身携带离线知识库（含勘探设备手册、地质标准规范、安全操作规程）。遇到设备故障或需要查阅技术规范时，即时提问获取答案。

场景三：应急救援现场信息查询

救援人员在灾害现场使用 APP 查询救援设备操作规范、急救流程等信息。现场通常无网络覆盖，且需要快速精确的信息检索。

场景四：户外运动领队行程保障

户外领队在带队前导入相关路线指南、安全手册、天气预报判读指南等资料。行程中遇到问题可随时查询，确保团队安全。


5. 系统架构
==========

5.1 架构分层
----------

表现层：Android Jetpack Compose + Material 3
- SplashScreen：启动引导与加载进度
- HomeScreen：底部Tab导航（诊断/手册）
- ChatScreen：对话界面
- ManualScreen：知识库管理界面

应用层：Kotlin
- ChatRepository：对话记录管理（Room 数据库）
- Retriever：RAG 编排（文档处理、向量索引、检索）

推理层：Kotlin + JNI
- ModelManager：模型解压、加载、生命周期管理
- InferenceService：Prompt 构建、输出清洗、免责声明

原生层：C++ (NDK)
- llama-jni.cpp：JNI 桥接
- llama.cpp：Qwen3.5-2B 推理引擎
- BGE-small-zh：Embedding 模型

5.2 数据流
----------

导入流程：PDF/TXT -> 文本提取 -> 句子切分 -> 重叠分块 -> BGE Embedding -> 索引持久化

问答流程：用户输入 -> BGE Query Embedding -> 余弦相似度检索 -> Top-5 分块 -> 句子窗口扩展 -> 模型推理 -> 输出清洗 -> 展示

5.3 技术栈
----------

| 层级 | 技术 | 版本 |
|------|------|------|
| 推理引擎 | llama.cpp | master |
| 生成模型 | Qwen3.5-2B Q4_K_M GGUF | 2B |
| Embedding | BGE-small-zh v1.5 Q4_K_M | 512维 |
| UI | Jetpack Compose + Material 3 | BOM 2024.04 |
| 本地存储 | Room (SQLite) + 自定义二进制格式 | Room 2.6.1 |
| 最低 SDK | Android 8.1 (API 26) | targetSdk 34 |
| 构建 | Gradle 8.5 + AGP 8.2.2 + CMake 3.22.1 | NDK 26.x |

6.2 需求详情
----------

F01-F03：启动流程

应用启动后自动加载本地模型。首次使用时将 GGUF 模型从 assets 解压到应用私有目录，后续启动直接加载已解压文件。

状态流转：IDLE 进入 EXTRACTING（解压中）进入 LOADING（加载中）进入 READY（就绪）或 ERROR（错误）

验收标准

| 验收项 | 要求 |
|--------|------|
| 首次启动耗时 | 解压模型至就绪，不超过 40 秒 |
| 冷启动耗时 | 已解压模型加载至就绪，不超过 5 秒 |
| 错误处理 | 加载失败时展示错误原因和重试按钮 |

F04-F08：知识库管理

用户通过系统文件选择器导入 PDF 或 TXT 文件，系统自动提取文本、切分句子、构建重叠分块、生成向量索引。

验收标准

| 验收项 | 要求 |
|--------|------|
| 格式支持 | PDF 和 TXT |
| 导入进度 | 分三阶段：文本提取（5%-20%）、句子切分（20%-35%）、向量索引（35%-100%） |
| 导入结果 | 显示文件名称和分块数量 |
| 文件管理 | 支持长按多选删除，删除前需确认 |
| 前置检查 | Embedding 模型未就绪时提示用户 |

限制条件

| 限制项 | 说明 |
|--------|------|
| 文档长度上限 | 单文档最多 2000 句 |
| PDF 类型 | 仅支持文字型 PDF，不支持扫描件 OCR |
| TXT 编码 | 优先 UTF-8，回退 GBK |

F09-F14：问答

用户输入问题后，系统检索知识库相关内容，基于 RAG 生成带原文引用的回答。

验收标准

| 验收项 | 要求 |
|--------|------|
| 有知识库 | 基于参考资料回答，附加来源编号 |
| 无知识库 | 提示请先导入手册文档 |
| 未找到内容 | 回复未找到相关信息 |
| 免责声明 | 回答末尾自动追加 |
| Think 标签 | 输出中的推理过程标签自动清理 |


7. AI 模型行为规范
============

7.1 模型角色定义
----------

| 维度 | 定义 |
|------|------|
| 角色 | 文档检索机器人 |
| 能力边界 | 仅能回答知识库手册中包含的内容 |
| 禁止行为 | 使用外部知识、编造信息、提供个人建议 |
| 输出语言 | 中文 |

7.2 推理参数
----------

| 参数 | 值 |
|------|-----|
| 生成模型 | Qwen3.5-2B Q4_K_M |
| Embedding 模型 | BGE-small-zh v1.5 Q4_K_M |
| 上下文窗口 | 2048 tokens |
| 最大生成 tokens | 1024 |
| 采样策略 | Greedy |
| CPU 线程数 | 自动检测（当前小米14测试机 8 线程中的 7 个） |

7.3 已知失败模式及应对
-------------

| 失败模式 | 表现 | 应对措施 |
|---------|------|---------|
| Think 标签溢出 | 模型输出 `<think>` 推理过程消耗 token 预算 | 正则检测并清理 think 块 |
| Think 标签不闭合 | `<think>` 无对应 `</think>` | 回退提取答案标记或保留生成内容尾部 |
| 回答偏离参考资料 | 模型使用训练知识而非文档内容 | 强制 system prompt + 参考资料置于 user 消息 |
| 输出格式偏离 | 未按要求输出来源引用 | 待优化（加强 prompt 示例） |
| Token 截断 | 超过 maxTokens 导致回答不完整 | 设置 1024 token 上限 |

7.4 提示词设计
----------

System Prompt：
```
你是严格的文档检索机器人。
规则：只使用【参考资料】中的原文回答。必须引用编号来源。
如果参考资料无相关内容，只回复：根据提供的资料，未找到相关信息。
直接给出最终答案。禁止思考推理过程。
```

User Message 格式：
```
【参考资料】
[1]《文档名》(位置百分比) 原文内容...
[2]《文档名》(位置百分比) 原文内容...

问题：{用户输入}
```


8. RAG 检索策略
===========

8.1 技术方案
----------

采用句子级重叠分块 + 窗口扩展检索策略。

8.2 分块策略
----------

原始文本按中文标点断句后，以 N 句为一组构建分块，相邻分块间重叠 M 句。

| 参数 | 值 | 说明 |
|------|----|------|
| CHUNK_SENTENCES | 3 | 每个分块包含 3 个句子 |
| OVERLAP_SENTENCES | 1 | 相邻分块重叠 1 个句子 |

示例（S 表示句子）：

| 分块 | 内容 | 说明 |
|------|------|------|
| Chunk 0 | S0 S1 S2 | 起始块 |
| Chunk 1 | S1 S2 S3 | 与 Chunk 0 重叠 S1、S2 |
| Chunk 2 | S2 S3 S4 | 与 Chunk 1 重叠 S2、S3 |

8.3 检索流程
----------

| 步骤 | 操作 | 说明 |
|------|------|------|
| 1 | 查询向量化 | BGE Query Embedding，加检索前缀指令 |
| 2 | 相似度计算 | 余弦相似度遍历全部分块 |
| 3 | Top-5 筛选 | 取相似度最高的 5 个分块 |
| 4 | 窗口扩展 | 每个命中分块前后各取 4 个句子 |
| 5 | 结果聚合 | 去重、拼接、格式化为参考资料 |

8.4 持久化格式
----------

| 文件 | 格式 | 内容 |
|------|------|------|
| chunks.json | JSON | 分块元数据（文本、句索引范围、来源文件） |
| sentences.json | JSON | 句子列表（用于窗口重建） |
| embeddings.bin | 二进制 | 浮点向量数组，5 字节头部（版本号+维度） |

|------|------|------|---------|
| 模型幻觉 | 中 | 高 | 强制引用来源、system prompt 约束、输出后校验 |
| 推理速度慢 | 高 | 中 | 当前 130 秒为物理限制，后续通过 GPU 加速改善 |
| APK 体积过大（1.4 GB） | 高 | 中 | 应用商店支持大文件上传，可考虑 AAB 格式 |
| 低内存设备崩溃 | 中 | 高 | 已启用 largeHeap，4 GB 设备建议降级 0.8B 模型 |
| PDF 解析失败 | 中 | 中 | 仅支持文字 PDF，不支持扫描件 |
| Embedding 模型加载失败 | 低 | 高 | 加载后执行 test embedding 验证 |
| 模型文件损坏 | 低 | 高 | 解压后校验文件大小 |
